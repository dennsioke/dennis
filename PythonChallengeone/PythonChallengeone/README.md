# Configurable URL Canonification Rules

Repository
https://github.com/karlicoss/promnesia
Commit: f102b0acd63b27a8053002c1e3de3cccd48c5adb

The built-in URL canonification rules cover a fixed set of well-known domains, but users working with private intranets, self-hosted tools, or niche services have no way to teach the library which query parameters matter for those domains. As a result, meaningful parameters get stripped and unrelated visits incorrectly collapse into the same normalised URL.
It would be useful to extend the Config with two optional fields. The first, EXTRA_SPECS, is a dict mapping domain strings to promnesia.cannon.Spec objects, following the same semantics as the built-in specs. When a custom Spec exists for a domain it fully replaces any built-in rule for that exact domain; a rule for wiki.corp.com does not apply to blog.corp.com. When qkeep is a list, the default params (id, t, p) are always included alongside any user-specified qkeep params, with user-specified params ordered after the defaults in definition order. When qkeep=True, all matched query parameters (including id, t, p) are kept and sorted alphabetically by key with no special precedence for the default params.

Domain key lookup for EXTRA_SPECS inherits the same suffix-matching semantics used by the built-in specs: a key is matched by iterating the domain's labels from the TLD upward and checking whether any accumulated suffix equals a key in the dict; the first (most-specific) match wins. Concretely, a key 'corp.com' matches 'corp.com', 'jira.corp.com', and 'dev.jira.corp.com', but does not match 'corp.net' (different TLD) or 'corp.com.au' (different suffix). A key consisting of a single label such as 'corp' matches only domains whose TLD is that exact label (e.g. 'issues.corp') and not domains where that label appears elsewhere in the hierarchy. Additionally, the prefixes 'www.' and 'amp.' are stripped from the domain before any spec lookup, so a key 'www.example.com' is permanently unreachable; the correct key for that domain is 'example.com'. The second field, EXTRA_DOMAIN_SUBSTITUTIONS, is a list of (from_prefix, to_prefix) string tuples applied as prefix substitutions to the domain portion of the URL only, before spec lookup. The list is scanned from first to last; the first entry whose prefix matches the domain fires and the scan stops immediately (first-match, definition-order semantics). A later entry with a longer or more-specific prefix is therefore unreachable if an earlier entry already matches; duplicate prefixes in the list only ever trigger the first occurrence. After the custom list is exhausted (or after one custom entry fires), the built-in substitution list is scanned in the same first-match fashion against the domain that resulted from the custom step. This means a custom substitution can chain into a built-in one: if the custom entry rewrites the domain to something the built-in list recognises, the built-in mapping also fires.
A mapping and a per-domain rule compose naturally: after a domain is remapped, the Spec for the resulting domain applies. Setting either field to an empty value should be a no-op. The rules must take effect immediately when config is set and must not persist once config is removed. Changes must propagate through the full indexing pipeline so that visits reflect the active config during canonification.
